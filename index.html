웹 개발에서 HTML은 단순한 마크업 언어가 아니라, 사용자 경험(UX)과 접근성(A11Y), 검색 최적화(SEO)를 결정하는 가장 중요한 요소라고 생각합니다. HTML을 단순히 페이지를 구조화하는 도구로 사용하는 것이 아니라, 적절한 태그를 사용하여 의미 있는 구조를 구성하는 것이 웹 개발의 핵심이라고 믿습니다. 저는 웹 표준을 준수하고, 시멘틱 태그를 활용하여 접근성과 유지보수성을 고려한 HTML을 작성하는 데 익숙하면서도, HTML이 단순한 기술이 아니라, 더 나은 웹 환경을 만들기 위한 설계 과정이라는 점을 인지하고 있습니다. 그렇기 때문에 HTML의 방대한 영역과 깊이를 고려했을 때, 제 실력을 85%로 평가하며, 지속적인 성장과 개선이 필요하다고 생각합니다. HTML을 활용하여 할 수 있는 것 - 웹 표준을 준수한 시멘틱 마크업 작성 - 웹 접근성을 위한 ARIA 속성 활용 및 스크린리더 대응 - 검색 엔진 최적화를 고려한 HTML 구조 설계 - 유지보수가 쉬운 HTML 설계 및 CSS, JavaScript와의 최적화된 연결 - 반응형 웹을 고려한 유연한 HTML 구조 구성
CSS는 시간이 지날수록 복잡해지기 쉽기 때문에, 재사용성을 높이고, 가독성을 유지할 수 있는 설계가 필수적이라고 생각합니다. BEM 네이밍, CSS 변수, 최소한의 코드로 효율적인 스타일링을 구축하는 것이 목표입니다. 모든 화면 크기에서 균형 잡힌 UI를 유지하는 것이 중요하다고 생각하기 때문에, Flexbox와 Grid를 적극적으로 활용하여 유연한 레이아웃을 설계하고 있습니다. 추가로 단순히 정적인 화면이 아니라, 사용자가 직관적으로 이해할 수 있는 자연스러운 동작을 추구하고, GSAP과 CSS 트랜지션을 활용해 부드러운 인터랙션을 구현하는 데 집중하고 있습니다. 디자인 시스템 구축, 성능 최적화, 애니메이션 연구, BEM 구조를 체계적으로 적용하여 더욱 발전하는 웹 퍼블리셔가 되는 것이 목표입니다. CSS로 할 수 있는 것 - CSS 변수 활용으로 유지보수성 강화 - 웹표준을 준수한 스타일 설계 (Flexbox, Grid 활용) - 반응형 웹 디자인(Media Queries & 유동적 레이아웃 적용) - GSAP 및 CSS 애니메이션을 통한 인터랙션 구현
자바스크립트는 웹의 동적인 요소를 제어할 뿐만 아니라, 사용자 경험을 극대화하고, 데이터를 다루며, 사고의 확장을 만들어주는 핵심 기술이라고 생각합니다. 저는 웹 퍼블리셔로서 HTML과 CSS를 넘어 JavaScript까지 활용할 수 있는 역량을 갖추는 것이 중요하다고 판단했고, 그 과정에서 자바스크립트의 방대한 영역을 체감했기 때문에 스스로 45%라고 평가했습니다. 하지만 이 45%는 제한적인 수치가 아니라, 앞으로의 성장을 위한 출발점입니다. 저는 JavaScript에서 코드 실행 순서를 고민하는 과정을 즐기며, 어떤 코드가 먼저 실행되고, 어떤 값이 나오는지를 예측하고 확인하는 과정에서 자바스크립트의 흐름을 이해하는 것이 중요하다고 생각합니다. 이러한 사고력은 단순한 스크립트 작성이 아니라, 웹 퍼블리셔로서 더 직관적인 인터랙션을 설계하고 디버깅을 할 때 논리적인 문제 해결 능력을 높이는 데 도움이 된다고 확신합니다. 현재는 웹 퍼블리셔로서 단순한 이벤트 핸들링을 넘어, 배열과 반복문을 활용하여 더 효율적인 코드 작성을 고민하는 단계로 나아가고 있습니다. JavaScript로 할 수 있는 것 - forEach, map 등 배열 메서드를 활용한 효율적인 데이터 처리의 능력을 빠르게 향상시키는 중 - 웹 퍼블리셔로서 필수적인 DOM 조작 및 이벤트 핸들링 구현 - 디버깅 과정에서 문제를 분석하고 논리적으로 해결하는 사고력 발전 - GSAP 및 CSS와 연동한 인터랙티브 애니메이션 효과
저는 jQuery가 DOM 요소에 직접 접근하지 않는 방식 때문에, 자바스크립트의 퍼포먼스 최적화 측면에서 비효율적일 수 있다는 점을 인지하고 있으며, 현재는 바닐라 JavaScript가 강력해졌고, React 같은 프레임워크가 대세가 된 상황이지만, 여전히 기존 레거시 프로젝트에서 jQuery를 사용하는 환경은 많은 것을 알고 있기 때문에, 이를 다룰 줄 아는 것이 웹 퍼블리셔로서 유용할 수 있다고 생각합니다. 그렇기 때문에, 저는 jQuery를 깊이 공부하지 않았지만, 기본적인 사용법을 알고 있으며, 필요할 경우 빠르게 익히고 적용할 수 있는 유연한 태도를 가지고 있습니다. 추가로 jQuery를 사용하면서 코드가 얼마나 줄어드는지 직접 체감했으며, 기본적인 DOM 조작이나 애니메이션을 구현할 때, jQuery의 체이닝 기법을 활용하면 코드가 훨씬 짧고 직관적으로 변한다는 것을 알고 있습니다. jQuery를 통해 경험한 것 - 이벤트 핸들링, 애니메이션 효과 사용 가능 - jQuery를 이용한 프로젝트 진행 경험 - 기본적인 JavaScript 문법을 jQuery로 변환해보며 코드 최적화 경험 - jQuery의 체이닝 기법을 활용하여 코드 길이를 줄이는 효과 체감
GSAP(GreenSock Animation Platform)는 단순한 애니메이션 라이브러리가 아닌. CSS 애니메이션보다 더 강력하고, 더 세밀한 컨트롤이 가능하며, 퍼포먼스 최적화까지 고려된 강력한 도구라고 생각됩니다. 저는 GSAP을 활용해 웹사이트의 인터랙션을 더욱 직관적이고 자연스럽게 만들고자 하고, 현재 기본적인 애니메이션 및 타임라인 개념을 이해하고 있으며, 이전 프로젝트에서도 GSAP을 활용한 다양한 모션 효과를 구현해 본 경험이 있습니다. 추가로 GSAP은 단순한 애니메이션 효과가 아니라 사용자의 행동을 유도하고, 더 나은 경험을 제공하는 인터랙션 설계의 일부이기 때문에, 더 깊이 있는 연구와 실전 경험이 필요하다고 생각합니다. GSAP을 활용하여 할 수 있는 것 - 기본적인 트윈 애니메이션 (Timeline 활용) - 스크롤 연동 애니메이션 (ScrollTrigger 활용) - SVG 애니메이션 및 Path Motion 적용 - CSS 애니메이션과의 비교 및 성능 최적화 이해 - GSAP의 성능 최적화를 이해하고, 불필요한 애니메이션을 최소화하는 사고 방식
저의 현재 Photoshop 실력이 30%로 부족하지만, 그 중요성을 누구보다도 인지하고 있으며, 지속적인 학습과 실전 적용을 통해 발전하고 있습니다. 디자인 툴을 자유롭게 다루는 것이 웹 퍼블리셔로서 필수적인 역량이라는 점을 인식하고 있으며, 디자인의 본질을 이해하는 것이 더 나은 UI를 만들기 위한 중요한 과정이라는 사실을 체감하고 있습니다. UI/UX 측면에서 디자인의 역할을 깊이 이해하려 노력 중이며, 퍼블리셔로서 디자이너와의 협업을 원활하게 하기 위해 Photoshop을 지속적으로 학습할 계획이고, 현재는 부족하지만, 빠르게 배우고 적용할 수 있는 역량을 갖추고 있습니다. Photoshop으로 할 수 있는 것 - 웹 디자인에 필요한 이미지 최적화 및 포맷 변환 - 기본적인 컷 편집 및 색상 보정 - 배경 제거(누끼 따기) 및 레이어 마스크 활용 - 그라디언트 적용 및 간단한 텍스트 효과 제작
저는 웹에서 벡터 기반의 요소(SVG)의 중요성을 인식하고 있으며, Illustrator를 활용하면 이를 더 정밀하게 컨트롤할 수 있다는 점을 경험을 통해 체감했습니다. 추가로 Illustrator는 단순한 그래픽 디자인 툴이 아니라, 웹 퍼블리싱에서 벡터 기반의 요소를 최적화하고, 인터랙션을 극대화할 수 있는 중요한 도구라고 생각합니다. 저는 현재 기본적인 벡터 제작과 SVG 활용을 할 수 있는 수준이지만, 빠르게 발전하여 UI/UX를 고려한 디자인을 만들고, 웹 퍼블리싱에 최적화된 벡터 작업이 가능하도록 성장하는게 계획입니다. Illustrator는 Photoshop보다 다룰 줄 아는 기능이 많지만, 여전히 디자인 역량을 보완해야 하며, 이를 빠르게 성장시키기 위한 학습을 멈추지 않을 계획입니다. 이러한 이유로 웹 퍼블리셔로서 Illustrator는 단순한 디자인 툴이 아니라, 웹 최적화를 고려한 벡터 제작 도구로 접근하고 있습니다. 일러스트 결론 - 디자인 감각과 벡터 활용 능력을 빠르게 끌어올리기 위해 적극적으로 학습하고 있음. - 현재 Illustrator 실력은 40% 수준이지만, 웹 퍼블리싱과 밀접한 벡터 개념을 이해하고 활용할 수 있음. - SVG 기반의 UI 요소를 제작하며, GSAP와 연계한 인터랙티브 애니메이션을 구현 가능. - 웹 퍼블리셔로서 Illustrator를 단순한 디자인 툴이 아닌, 웹 최적화 도구로 접근하고 있음.
저는 디자인 초안을 작성한 후 웹을 만드는 것과, 아무런 가이드 없이 바로 개발하는 것의 차이를 프로젝트를 통해 강하게 체감했으며, 특히 와이어프레임 단계의 중요성을 경험하면서, 디자인을 먼저 구상하고 정리하는 과정이 얼마나 개발의 효율성을 높이는지 직접 확인했습니다. 이전에는 바로 코드 작업에 들어가는 경우가 많았지만, Figma를 활용하여 먼저 구조를 고민하고 초안을 만들어보는 것이 훨씬 더 체계적이고 논리적인 개발 프로세스를 만든다는 점을 깨달았습니다. 현재 Figma를 매우 최근에 배웠기 때문에, 기본적인 컴포넌트 개념을 이해하는 수준이지만, 저는 웹 개발을 할 때 항상 여러 변수를 고려하는 사고방식을 가지고 있고, 이 사고방식이 Figma에서도 그대로 적용될 수 있다고 판단하고 있습니다. 즉, 저는 퍼블리셔로서 단순히 Figma를 배우는 것이 아니라, 이를 활용해 더 논리적이고 효율적인 UI 설계를 할 수 있도록 발전해 나갈 계획입니다. 컨셉 : 러너들의 교류, 러닝 정보 공유 및 매칭을 위한 자체 콘텐츠 기반 커뮤니티 사이트

기술 스택 : HTML / CSS

디자인 스택 : Adobe Illustrator / Photoshop

간단 설명

프로젝트 오런완은 2인 팀 프로젝트로, HTML/CSS만을 이용하여 정적인 데스크탑 웹사이트를 설계 및 구현하는 것이 목표였습니다.

단순한 웹사이트 제작이 아니라, 웹 설계 단계에서 기획된 내용을 HTML/CSS 구조로 정확하게 표현하는 것에 중점을 두었습니다.

또한, 이 프로젝트를 통해 웹의 기본적인 동작 원리를 이해하고, HTML/CSS의 시맨틱 구조와 올바른 코드 작성 방식을 학습하는 데 집중했습니다.

특히, 2인 협업을 통해 HTML/CSS 코드 스타일을 맞추고, 웹사이트 설계 의도를 반영하는 과정에서 팀워크와 코드 협업의 중요성을 배운 프로젝트였습니다.

문제 & 해결 과정
➥ 애플 애니메이션 구현 문제
• 애플 애니메이션처럼, 여러 스크롤 기반 애니메이션을 동시다발적으로 구현하면서, 동시에 slide되는 요소의 scale되는 트리거도 계속 바뀌는 것을 구현하고자 함
› 기존 script에만 의존하던 방식에서, css sticky 속성과 margin-top의 음수설정을 활용해서 아래에서 위로 걷히는 부분 구현함
› 동시에 gsap 스크롤 트리거를 위해, tabindex와 aria-hidden을 적용한 트리거 전용 div를 만들어서 스크롤 트리거를 다채롭게 적용함
› 각 swiper에 scale이 커지게끔 absolute로 띄운 img들을 하나 더 배치하고, not 선택자를 사용해 display none을 해서, 각 swiper의 active 클래스가 있는 element만 scale이 커지게 적용, 이로 인해 swiper active가 바뀌어도 가운데 요소만 커지는 것을 구현함
※ 개선 방향: swiper의 MutationObserver와 GSAP에서 killAnimation, 트리거 새로고침 등을 활용하면, trick을 쓰지 않고 swipe마다 새로고침하는 방식으로 개선할 수 있을 것으로 보임. 하지만, 현재 실력 부족으로 위 방식으로 해결함.
➥ 많은 애니메이션 사용으로 인한 느린 로딩 문제와 그에 따른 사용자 경험
• GSAP 애니메이션, 비디오, 고해상도 이미지 사용으로 인해 로딩 속도가 느려졌으며, 페이지가 완전히 로드되지 않은 상태에서 애니메이션이 실행되면 의도한 효과가 나오지 않는 문제가 발생함.
› 첫 번째 해결책으로, Pre-loader를 도입함. 스크롤 기반 애니메이션을 만들 때 Pre-loader는 필수적이라고 판단함.
› 프리로더는 문서 내 모든 이미지를 검색한 후, 총 이미지 개수를 변수에 저장하고, 현재 로드된 이미지 개수를 기준으로 퍼센트를 계산하여 진행도를 표시함. GSAP의 `onUpdate()`를 활용하여 `textContent` 값이 100%까지 부드럽게 증가하도록 구현. 로딩이 완료되면 `callback()`을 통해 프리로더를 제거하고 페이지의 모든 요소가 정상적으로 작동하도록 처리함.
› 로드 속도 최적화를 위해 리소스 용량을 줄이는 작업을 진행함. PNG를 SVG 및 AVIF로 변환하고, CSS/JS Minify 적용, 폰트 CDN을 `preconnect`로 미리 연결하여 성능을 개선함.
※ 배운 점: GSAP을 과도하게 사용하는 것보다 Vanilla JS와 CSS 속성을 적극 활용하는 것이 성능 저하를 방지하는 데 효과적임. 또한, `width/height/left/top` 변경보다는 `transform`과 `opacity`를 활용하는 것이 렌더링 성능을 최적화하는 데 더 유리함.
➥ 웹 접근성과 검색엔진 최적화, 그리고 UX 개선
• 웹표준을 준수하며, 웹 접근성, SEO 최적화, UX 개선의 중요성을 크게 깨달았고, 이에 초점을 맞춰 프로젝트를 진행함.
› 접근성을 높이기 위해 `aria-hidden`, `alt` 속성, `tabindex="-1"` 등을 활용하여 스크린 리더에서 불필요한 요소를 감춤. 또한 배경 비디오에는 `role="presentation"`을 추가하고, 탭이동(`tab`)이 가능한 요소에 `focus` 스타일을 적용하여 키보드 사용자 경험을 개선함.
› 버튼 클릭 시 열리는 Modal 창에서는 `ESC` 키를 누르면 자동으로 닫히도록 기능을 추가하여, 사용자의 편의성을 높임.
› 다크모드 & 라이트모드 환경에 맞춘 파비콘을 제작하고, 구형 브라우저에서도 지원되는 대체 파비콘을 적용. 또한 애플 홈 화면 아이콘(iOS)도 다크/라이트 모드에 따라 각각 제작하여 최적의 사용자 경험을 제공함.
› SEO 최적화를 위해 `meta name="description"`을 작성하고, Open Graph 및 Twitter Card 메타 태그를 추가하여, 페이지가 SNS에서 공유될 때 최적화된 썸네일과 함께 미리보기가 잘 표시되도록 구성함.
※ 개선 사항: 모달이 열렸을 때 `tabindex`를 모달 내부로 제한하고, 다른 모달로 이동할 때 `tabindex`를 동적으로 변경하는 기능을 구현하려 했으나, 완벽하게 작동하지 않아 결국 프로젝트에서 제외함. 하지만 이 기능은 웹 접근성과 UX에서 필수적인 요소이며, 반드시 개선이 필요하다고 판단함. 따라서 향후 `focus-trap` 또는 `aria-hidden` 속성을 활용한 보다 완벽한 구현 방안을 연구할 계획.
➥ 폴더 구조 정리와 유지보수를 위한 CSS/JS 파일 분리, 함수의 모듈화
• 프로젝트의 폴더 구조, CSS/JS 파일의 기능별 분리, 함수 모듈화가 유지보수성과 코드 확장성에 큰 영향을 미친다는 점을 깨닫고 이를 적극적으로 적용함.
› 프로젝트 폴더를 `src / public / assets / docs` 4개의 카테고리로 나누고, 내부에서도 세부적으로 분류하여 체계적인 구조를 설계함.
› JS와 CSS는 각각 용도별로 세분화하고, JS는 기능별 모듈화하여 `main.js`에서 전체적인 흐름을 제어하도록 구성함.
› JS 내부에서도 하나의 함수에 너무 많은 기능이 포함되지 않도록 분리하여, 수정이 용이하도록 모듈화에 신경 씀.
› CSS에서는 `:root` 변수를 활용해 기획 단계에서 정한 폰트, 컬러, 간격 값을 변수로 저장하고, 자주 사용되는 스타일은 전역 클래스로 만들어 일관성을 유지함.
› 또한, HTML 내에서 CSS와 JS를 최적화된 로딩 순서에 맞춰 배치하고, `defer` 속성을 활용하여 불필요한 렌더링 지연을 최소화함.
※ 개선 사항: CSS에서 사용하는 변수명과 JS에서의 변수, 함수명은 그 목적과 역할이 직관적으로 전달될 수 있도록 명확하게 작성해야 한다는 점을 이번 프로젝트를 통해 절실히 깨달음. 하지만 이 부분에서 미흡한 점이 있었고, 특히 BEM(Block Element Modifier) 네이밍 체계에 대한 이해가 부족했던 점이 아쉬움으로 남음.
추가로 네이밍 규칙이 체계적으로 정리되어 있다면, 코드만 보더라도 개발자 간의 협업이 쉬워지고, 프로젝트를 처음 접하는 사람도 설명 없이 의도를 파악할 수 있기 때문에 코드의 유지보수성이 극대화될 것이라고 판단함.
또한, 이번 경험을 바탕으로 BEM 네이밍을 더욱 체계적으로 적용하는 방법을 연구하고, 변수명과 함수명을 설계할 때 의도를 명확하게 드러내는 습관을 길러야겠다고 다짐함.

문제 & 해결 과정
➥ 문서 정리와 주석의 중요성 - 유지보수
• 첫 반응형 프로젝트 & 첫 스크립트 적용 프로젝트였던 만큼, 수많은 구조 수정 과정에서 문서 정리 미흡이 유지보수에 미치는 영향을 직접 체감하며, 함수 모듈화에 대한 중요성까지 인식
› 기존에는 단순히 `html`, `css`, `js`만 폴더별로 보관했으나, 이 프로젝트를 계기로 `image` 폴더 내에서도 각 섹션 & 기능별로 체계적으로 파일을 분리하는 방식을 적용
› 주석을 제대로 작성하지 않으면 코드 수정 시 엄청난 비효율이 발생한다는 것을 경험. 특히 jQuery를 활용한 애니메이션 및 인터랙션 코드에서 나중에 수정하려고 보면 어디가 어떤 기능인지 파악하는 데 시간이 오래 걸림을 직접 체감
› 이를 해결하기 위해 기능별 & 섹션별로 필요한 부분에는 주석을 철저히 작성하는 습관을 들이기 시작
※ 얻어간 것: 이 경험을 바탕으로 3차 프로젝트(CLAYN)에서는 기획 단계부터 체계적인 폴더 구조 & 카테고리화, 주석 및 문서 정리를 철저히 설계하는 방법을 배울 수 있었음. 또한, 복잡한 스크립트 코드나, 각 섹션에는 표시나 설명을 위한 주석을 추가해 가독성과 유지보수성을 극대화하는 방법을 배움.
➥ 사이트 이탈률과 연관한 인터랙티브의 중요성
• 사용자는 스크롤이 반복될수록 흥미를 잃고, 이는 곧 사이트 이탈률 증가로 이어질 수 있음. 따라서, 단순한 나열형 콘텐츠가 아니라 사용자가 더 직관적으로 정보를 접할 수 있도록 UI/UX 설계를 고민함.
› `jQuery`의 `.each()` 메서드와 `JavaScript`의 `getBoundingClientRect()` 값을 활용하여, 요소의 `bottom - top` 값을 계산해 스크롤 위치에 따라 opacity를 동적으로 조절하는 애니메이션 효과 구현
› 단순 일렬 나열이 아닌, - 사용자의 목적에 맞는 `tab` 기능을 활용하여 콘텐츠를 정리하고, - '더보기' 버튼을 통해 필요한 정보만 추가적으로 제공, - Flexbox를 활용해 한눈에 정보가 들어오도록 배치하여 스크롤 부담을 최소화
※ 배운 점: 웹이 단순히 디자인 요소를 꾸미는 것이 아니라, 정보 전달의 역할을 한다는 점을 깊이 깨달음. 특히, 이탈률을 줄이고 사용자의 흥미를 유지하려면, UX 설계와 MD적인 콘텐츠 배치 전략이 결합되어야 한다는 점을 인지하게 됨.
➥ resize의 성능 문제에 따른 새로고침 delay와, 모바일 스크롤에서의 문제점
• 사용자의 브라우저 크기 변화를 감지하여, 스크립트를 초기화하는 과정이 필요함을 인지. 하지만 단순히 `resize` 이벤트를 감지할 경우, 모든 `resize` 이벤트가 지속적으로 실행되면서 성능 저하 문제가 발생
• 또한, 모바일에서는 스크롤 시 브라우저 주소창이 자동으로 사라지거나 나타나면서 스크롤할 때마다 `resize` 이벤트가 발생하는 문제를 발견
› resize 성능 문제 해결: - `setTimeout` + `clearTimeout`을 활용한 debounce 기법 적용 - 마지막 `resize` 이벤트가 실행된 후 일정 시간(delay)이 지나야 새로고침이 실행되도록 설정
› 모바일 스크롤 시 resize 문제 해결: - `resize` 이벤트에서 width 변화만 감지하도록 조건 추가 - 이를 통해 브라우저의 주소창으로 인한 높이(height) 변화는 무시하도록 설정
※ 배운 점: 이 경험을 통해 JavaScript에서 "이벤트 트리거(trigger)"가 언제 발생하고, 트리거를 언제 발생시키는지에 대한 개념의 인지를 함.
또한, 성능 최적화를 위해 이벤트 리스너를 무조건 실행하는 것이 아니라, debounce 같은 기법을 활용해 최소한으로 실행해야 한다는 점을 확실히 인지
➥ HTML 스케일링 문제
• HTML의 구조가 지나치게 비대해지는 문제가 발생했으며, 콘텐츠가 추가될수록 HTML 코드의 개수도 증가하는 문제가 있었음.
• `map`, `filter`, `find`, `includes`, `some`, `every` 등의 JavaScript 메서드에 익숙하지 않았고, 이를 대체하기 위해 중첩 if문과 직접 HTML을 추가하는 방식을 사용
• 이로 인해, - 중첩된 `if`문이 많아질수록 가독성이 떨어졌고 - 특정 변수마다 다르게 적용할 HTML이 필요해지면서 - 결과적으로 HTML 파일의 크기가 지나치게 커지는 문제가 발생함
※ 아쉬운 점:
`if`문을 활용한 해결책은 나름대로 유효했지만, 그 이전에 `map`, `filter`, `find`, `includes` 등의 메서드를 먼저 학습한 이후에 적용했다면 더 효율적으로 HTML을 줄일 기회였다는 점이, 개인적으로 아쉬움.
※ 배운 점:
어떻게 해서든 구현하기 위해`if`문으로 sort 기능을 구현하는 과정에서, 스크립트를 설계하는 사고력이 크게 향상되었고, 이 경험이 이후 프로젝트에서도 큰 도움이 되었음.
※ React 없이도 HTML 최적화는 최대한 해야 한다는 점을 인지:
JS의 Web Component를 활용하면 HTML을 컴포넌트 단위로 관리할 수 있었을 것이라는 점이 아쉬움으로 남았기에- 향후, JavaScript 기반으로 HTML을 더 동적으로 관리하는 방법을 연구할 계획
※ 이 경험을 통해 얻은 가장 중요한 교훈: 비대해진 HTML 구조는 성능과 유지보수에 직접적인 영향을 미친다는 점을 체감했으며, 코드 최적화와 성능 문제를 항상 염두에 두는 계기가 된 프로젝트였음.

주요 고려 사항
• HTML/CSS를 활용하여 기획한 내용을 데스크탑 웹페이지로 구현
• 웹 콘텐츠를 구성할 때, 사용자가 요소의 의미를 직관적으로 이해할 수 있도록 UI/UX 설계
• HTML 구조와 CSS 스타일링에서 웹페이지의 필수적인 기본 요소(시맨틱 마크업, 레이아웃, 접근성 등)를 놓치지 않고 반영
개인적 목표
• HTML/CSS에 대해 1개월 동안 학습한 내용을 바탕으로, 웹 레퍼런스 없이 코딩으로 구현하여 기본기를 다지는 것이 목표
• CSS 스킬을 향상시키기 위해 선택자 조합, 상태 기반 스타일링, 가상 요소 활용 등 다양한 CSS 기술 적극 활용
• 여러 `input` 타입을 사용해보면서 폼 요소의 다양한 동작 방식과 `display` 속성의 상태 변화를 깊이 이해
문제 & 해결 과정
➥ 첫 웹페이지 제작에서의 고민과 도전
• 웹페이지 제작에 있어 디자인과 코딩을 모두 직접 고민하며, 기획한 내용을 설계하고 구현하는 데 집중
• 처음부터 모든 요소를 직접 생각하고 구현해야 했기 때문에, 필요한 개념을 그때그때 학습하면서 진행
※ 얻어간 것:
이 과정에서 실력이 비약적으로 상승하는 것을 체감하였고, 또한 "구현하면서 배우는 방식"이 효과적이라는 것을 깨달음. 이후 프로젝트에서도 이 방식을 적용하여 자발적으로 학습하는 습관을 기르게 됨.
➥ 첫 팀 프로젝트에서의 협업 방식 결정
• 개인 프로젝트가 아닌 팀 프로젝트였기 때문에, 작업 영역이 겹치거나, 동일한 요소에 서로 다른 클래스가 적용되는 문제가 발생
› 공통 요소에 대한 `ClassName`을 사전에 정의하고, 이 네이밍 규칙을 변경하지 않도록 가이드라인을 설정
› 공통 `ClassName`에 대한 스타일을 CSS 파일 상단에 배치하여, 협업 시 스타일 규칙을 유지할 수 있도록 정리
› ID 속성을 활용해 각 HTML 파일에서 개별적으로 스타일을 적용할 수 있도록 하고, CSS에서 ID의 자식 선택자를 이용하여 스타일 충돌을 방지
› 역할 분배를 명확하게 정리하고, 팀원들의 강점과 약점을 파악하여 최적의 업무 분배를 진행
※ 배운 점: 팀 프로젝트에서는 단순히 각자 작업하는 것이 아니라, 사전에 명확한 규칙을 정하고, 합의된 방식대로 작업하는 것이 필수적이라는 것을 깨달음. 이를 통해 코드 충돌을 줄이고, **협업이 원활하게 이루어지는 환경을 만드는 것이 중요하다는 것을 직접 경험**
➥ 웹 표준 및 웹 접근성 준수
• 웹 표준 검사(Web Validation)를 실행했을 때, `input` 태그에 `label`이 누락되어 있거나, `img` 태그에 `alt` 속성이 없는 등 여러 문제를 발견
• 이를 수정하는 과정에서 웹 표준과 접근성을 준수하기 위해 고려해야 할 요소들을 체계적으로 학습
※ 하나하나 다시 수정하며 웹 표준을 준수하기 위해 어떤 요소를 신경 써야 하는지 실전 경험을 통해 깨달음. 이후 모든 프로젝트에서 웹 접근성을 중요하게 고려하는 계기가 됨.
